# Как упаковать Python GUI-приложение в AppImage

Практическое руководство, написанное по итогам реальной упаковки Paperi
(customtkinter + tkinterdnd2 + PyMuPDF + Pillow, плюс отдельная головная боль
с apt-only скомпилированным расширением для Bluetooth). Ориентировано на
будущие Tkinter/customtkinter GUI-проекты, а не только на Paperi —
конкретный рабочий пример лежит в `packaging/appimage/` этого репозитория,
на него можно ссылаться из других проектов как на шаблон.

## Инструмент: PyInstaller, а не сырой python-appimage

Для Tk-based приложений — **PyInstaller** (`--onedir`), обёрнутый вручную в
AppDir + `appimagetool`. Не `python-appimage` (голый интерпретатор без
аккуратной обработки Tcl/Tk data-директорий — ловит `_tkinter.TclError:
invalid command name "::msgcat::mcmset"`, ошибку про ненайденные Tcl
init-скрипты) и не `linuxdeploy-plugin-python` (фактически не
поддерживается). PyInstaller уже 15+ лет умеет правильно находить и
копировать Tcl/Tk рантайм рядом с `_tkinter.so` — это единственная причина
всей затеи с ним, а не потому что PyInstaller как-то особенно хорош для
AppImage в целом.

`--onedir`, не `--onefile`: быстрее стартует (`--onefile` каждый раз
распаковывает себя во временную директорию при запуске), и это естественная
форма, которую оборачивает AppDir (`AppDir/usr/bin/<name>/` = ровно то, что
кладёт `COLLECT`).

## Специфичные для библиотек грабли (нашли на Paperi)

- **customtkinter**: темы/шрифты/иконки — package data (JSON, TTF), не код.
  Без `collect_data_files("customtkinter")` в spec-файле приложение либо не
  стартует, либо стартует с сырым/пустым UI. PyInstaller это НЕ находит
  автоматически.
- **tkinterdnd2**: несёт с собой предкомпилированную нативную библиотеку
  (`libtkdnd*.so`) и Tcl-пакет как package data, не только `.py`. Нужен
  `collect_all("tkinterdnd2")`, не только `collect_data_files`.
- **PyMuPDF**: импортируется и как `fitz`, и как `pymupdf` в зависимости от
  версии — на всякий случай собирайте оба через `collect_all`, обёрнутые в
  `try/except`, лишний собранный (или не найденный) пакет не ломает билд.
- Всё это надёжнее задавать в **`.spec`-файле** через
  `PyInstaller.utils.hooks.collect_data_files`/`collect_all`, а не через
  голые CLI-флаги `pyinstaller app.py` — иначе список `--collect-*` флагов
  расползается и его легко забыть при следующей правке.

## Билд-окружение: Docker с намеренно старым базовым образом

Собирайте внутри Docker (`ubuntu:22.04` или ещё старее), а не на
хост-машине напрямую — так итоговый AppImage не окажется случайно привязан
к glibc-версии именно вашей машины разработчика. Хост для сборки и хост для
запуска — разные вещи; собирать нужно на самом старом glibc, который вы
готовы поддерживать, а бежать AppImage будет там, где скажет пользователь.

Минимальный образ `ubuntu:22.04` **не содержит `gnupg`** — без него
`add-apt-repository` (например, для deadsnakes PPA, если системный Python
контейнера недостаточно свежий) падает на импорте GPG-ключа с
малопонятной ошибкой `ShortcutException`. Добавляйте `gnupg` в первый же
`apt-get install` явно, до любого `add-apt-repository`.

Минимальный образ также **не содержит базовых X11 runtime-библиотек**.
PyInstaller тихо (WARNING, не ошибка сборки) не смог зарезолвить
`libXcursor.so.1` — зависимость нативной библиотеки tkinterdnd2 — просто
потому что её не было в билд-контейнере. Итог: PyInstaller не может её
забандлить, и AppImage перестаёт быть по-настоящему самодостаточным именно
в той части, ради которой AppImage и затевался. Установите нужные X11-либы
(в нашем случае `libxcursor1`) явно в Dockerfile и **проверяйте лог сборки
на `WARNING: Library not found`** — PyInstaller не завалит билд из-за
недостающей system-lib, он просто молча соберёт AppImage похуже.

## Ключевой урок: glibc билд-контейнера ≠ glibc времени выполнения

Если приложение зависит от **системного** скомпилированного расширения
(апт-пакет, не pip wheel — в нашем случае `python3-bluez`), легко перепутать
две разные проблемы:

1. **Python ABI (версия интерпретатора)** — расширение `foo.cpython-312-
   x86_64-linux-gnu.so` попросту не подхватится интерпретатором другой
   минорной версии; это решается только совпадением минорной версии
   вшитого в AppImage интерпретатора с системным `python3` целевого
   дистрибутива.
2. **glibc-версия самого `.so`-файла** — расширение, собранное под
   свежий дистрибутив, может требовать симфолы вроде `GLIBC_2.38`,
   которых нет в вашем нарочно-старом билд-контейнере.

Тестировать это ВНУТРИ билд-контейнера (`docker run ... python3.12 -c
"import bluetooth"`, смонтировав системную dist-packages с хоста) даёт
ложноотрицательный результат по пункту 2 — контейнер специально собран на
старом glibc, а реальный AppImage при **настоящем** запуске использует
`ld.so`/`glibc` хост-системы пользователя, а не контейнера. Единственный
надёжный тест на пункт 2 — запустить сам собранный `.AppImage` на
реальной целевой машине. Docker-контейнер годится только для проверки
пункта 1 (и то с осторожностью — либо монтируйте туда и системные
`.so`-зависимости самого расширения, либо тестируйте по факту сообщения об
ошибке: `undefined symbol`/`wrong ELF class` = ABI, `version 'GLIBC_x.y' not
found` = версия glibc, это два разных класса ошибок с разными выводами).

## Проверка после сборки: чек-лист

1. **Смотрите `WARNING: Library not found` в логе PyInstaller** — самый
   дешёвый способ поймать недостающие runtime-либы билд-окружения раньше,
   чем это найдёт пользователь.
2. Запускайте итоговый `.AppImage` на реальной машине **без активированного
   venv** — это и есть сценарий, ради которого всё затевалось.
3. Явно тестируйте не только «окно открылось», но и функциональность,
   зависящую от package data (темы/шрифты — если бы `--collect-data
   customtkinter` забыли, GUI бы стартовал визуально «пустым», а не упал бы
   с трейсбеком) и от subprocess-вызовов системных бинарников
   (`bluetoothctl` и подобные — PyInstaller-бутлоадер модифицирует
   `LD_LIBRARY_PATH` перед запуском дочерних процессов, что теоретически
   может сломать системные бинарники; на практике для `bluetoothctl` не
   сломало, но не проверять — не вариант).
4. Если приложение зависит от системного апт-only компилируемого
   расширения (см. предыдущий раздел) — идеально протестировать реальный
   рантайм-путь через него живьём на целевой машине, а не только через
   косвенные признаки.

## Три стратегии для «апт-only компилируемого расширения без pip wheel»

Общий паттерн, не специфичный для Bluetooth — встречается для любых системных
библиотек без нормального manylinux-колеса:

- **(а) Подогнать минорную версию вшитого интерпретатора под целевой
  дистрибутив, документировать ограничение.** Меньше всего работы и риска.
  Ломается, если у пользователей разъезжаются версии системного Python
  (разные релизы дистрибутива). То, что сделали в Paperi — см.
  `infra/peripage_client.py::_default_printer_factory` — с явной, понятной
  ошибкой вместо падения, если версии не совпали.
- **(б) Скомпилировать расширение из исходников прямо в AppImage** на этапе
  сборки, используя тот же интерпретатор, что и весь остальной бандл —
  тогда ABI совпадает by construction, и это по-настоящему
  zero-dependency AppImage. Дороже: нужно воспроизводить любые патчи,
  которые накладывает пакет дистрибутива (апт-пакеты нередко патчат
  апстрим специально под текущий Python), и валидировать под каждый
  таргетируемый glibc.
- **(в) Изолировать за границей subprocess/IPC к системному `python3`.**
  Полностью устойчиво к разъезду версий интерпретатора ценой архитектурного
  усложнения (нужен протокол между процессами). Оправдано, когда (а) на
  практике оказывается недостаточно надёжным, а (б) слишком дорого
  поддерживать.

## Рабочий пример

`packaging/appimage/` в этом репозитории: `Dockerfile` (билд-окружение),
`paperi.spec` (PyInstaller spec с `collect_data`/`collect_all`),
`AppRun`/`paperi.desktop`/`paperi.png` (AppDir), `build.sh`
(оркестрирует всё end-to-end, включая скачивание `appimagetool`, которого
нигде заранее не должно быть установлено — воспроизводимость превыше
удобства «а у меня уже стоит»).
